## Commands

```bash
npm run dev          # Start dev server at http://localhost:3000
npm run build        # Production build
npm run start        # Serve production build locally
npm run lint         # ESLint (Next.js core-web-vitals + TypeScript)
npm run gen:types    # Regenerate src/types/database.ts from Supabase
npm run push:vapid   # Generate VAPID key pair for web push (one-time setup)
```

Run `npm run gen:types` after modifying SQL in `supabase/migrations/`. Requires `SUPABASE_PROJECT_ID` and `SUPABASE_ACCESS_TOKEN` in `.env.local`. Never hand-edit `src/types/database.ts`.

Pre-PR checks: `npm run lint` + successful `npm run build` (no automated test suite yet).

## Architecture

**AllNoodles** is a Next.js 16 App Router PWA for a noodle distribution business. UI is in Thai (Sarabun + IBM Plex Mono + Instrument Sans via Google Fonts; LINE Seed Sans TH local fonts in `src/app/fonts/` used in print layouts). Uses Supabase (PostgreSQL + RLS), LINE LIFF for mobile integration, and Tailwind CSS 4. React Compiler is enabled. Offline support via `/offline` page and service worker.

### Authentication

Custom PIN-based auth — no Supabase Auth:
- User enters 6-digit PIN → `src/app/login/actions.ts` calls `verifyPin()`
- PIN hashed with scrypt + pepper (`LOGIN_PIN_PEPPER`) and compared against `pin_lookup` table
- On success: HMAC-signed cookie (`allnoodles_session`) created via `src/lib/auth/session.ts`
- Session contains: user ID, display name, role (`admin | member`), org ID, expiry
- Middleware in `src/proxy.ts` protects `/dashboard/:path*` via `updateSession()` in `src/lib/supabase/middleware.ts`
- Use `requireAppSession()` from `src/lib/auth/authorization.ts` as an auth guard in server components/actions
- Use `requireAppRole("admin")` for admin-only routes (e.g. `/delivery`, `/orders/incoming`) — redirects non-admins to `/dashboard`

### Supabase Clients

Four distinct clients — use the right one:
- `src/lib/supabase/server.ts` → `createClient()` — for Server Components (read-only cookie access)
- `src/lib/supabase/action.ts` → `createActionClient()` — for Server Actions that need to write cookies
- `src/lib/supabase/browser.ts` — for Client Components
- `src/lib/supabase/admin.ts` → `getSupabaseAdmin()` — service role, bypasses RLS (dashboard aggregates, admin ops)

### Data Fetching Pattern

Prefer Server Components for reads, Server Actions (`"use server"`) for mutations. Keep sensitive logic in `src/lib` with `"server-only"` imports to prevent client-side leakage.

### Domain Model

Key entities (all org-scoped via `organization_id`, RLS-enforced):
- **`organizations`** — root tenant; `metadata` JSON stores org-level `address` and `phone` used in billing/delivery print headers
- **`customers`** — has `customer_code` (TYS-prefix auto-seq), `billing_day`, `default_vehicle_id`, `line_user_id`
- **`products`** — has `sku` (TYN-prefix auto-seq), `product_unit` enum (`kg|pack|bag|piece|tray`), `stock_quantity`, `reserved_quantity`
- **`product_sale_units`** — multiple sale units per product (label, `base_unit_quantity`, cost override); the primary unit is on `products` itself
- **`product_images`** — multiple images per product, sorted
- **`customer_product_prices`** — per-customer sale prices keyed by `(customer_id, product_sale_unit_id)`
- **`orders`** / **`order_items`** — status: `draft → submitted → confirmed → cancelled`; order number format `TY-YYYYMMDD-XXXX`
- **`inventory_receipts`** / **`inventory_movements`** — audit trail for stock; movement types: `receipt | reserve | issue | release | adjustment`
- **`vehicles`** — delivery vehicles, org-scoped
- **`delivery_notes`** — one per confirmed order; tracks dispatch status and item adjustments. Item quantities can be corrected post-confirmation via the `adjust_delivery_note_item(p_organization_id, p_delivery_note_item_id, p_new_quantity_delivered, p_adjusted_by, p_resolution_mode)` RPC — current resolution mode used is `"lost"`
- **`billing_records`** — locked billing statements generated from delivered orders; store a `snapshot_rows` JSON so past totals are immutable even if delivery notes are later edited

The order workboard (`/orders`) uses PostgreSQL RPC functions (`get_order_daily_store_summaries`, `get_order_daily_store_items`) defined in migrations — modify those RPCs when changing workboard query logic. Billing sequence numbers are generated by the `next_billing_number(p_organization_id, p_billing_date)` RPC.

**Product sale unit cost modes**: each `product_sale_units` row has a `cost_mode` of `"derived"` (cost = base price × `base_unit_quantity`) or `"fixed"` (explicit `fixed_cost_price`). Use `getEffectiveSaleUnitCost()` from `src/lib/products/sale-unit-cost.ts` wherever sale unit cost is needed — never reimplement this logic inline.

Key routes:
- `/order` — LINE LIFF customer-facing order placement (no PIN auth, identified by `line_user_id`)
- `/orders` — internal admin workboard (daily store summaries via RPC)
- `/orders/incoming` — flat list of all incoming orders by date, admin-only
- `/orders/delivery-notes/[id]` — printable delivery note for a confirmed order
- `/orders/delivery-notes/preview` — preview/print combined delivery note
- `/orders/packing-list` — printable product×store matrix for a given date (admin-only)
- `/delivery` — delivery dispatch board (tick off delivered orders), admin-only
- `/delivery/print` — printable delivery summary
- `/billing` — billing statement creation + history, admin-only
- `/billing/print` — preview/print a single billing statement; `?save=true` locks the snapshot
- `/dashboard` — overview KPIs and quick links, middleware-protected
- `/dashboard/settings` — legacy redirect → `/settings/products` (no longer a real page)
- `/stock` — stock overview; `/stock/movements` — movement history
- `/settings/customers` — customer CRUD; `/settings/customers/pricing` — per-customer sale price management
- `/settings/products` — product catalog management
- `/settings/stock` — stock receipts entry
- `/settings/vehicles` — vehicle management
- `/reports/sales-overview` — annual revenue/profit charts by month
- `/reports/product-sales` — product-level sales ranking with date range filter
- `/reports/store-sales` — store-level sales ranking with revenue/profit/margin
- `/reports/delivery-notes` — delivery note history and search
- `/api/geography` — cascading province/district/subdistrict lookup from `thailand-geography.json` (used by address forms; responses are cached for 1 day)
- `/api/order/session` — sets/reads the LIFF customer session cookie (`allnoodles_order_session`)

### Route Protection

Middleware (`src/proxy.ts`) only runs on `/dashboard/:path*` and `/login`. All other routes (including `/order`, `/orders`, `/settings`, `/stock`) are **not** protected by middleware — they rely on `requireAppSession()` called inside each server component/action individually.

### LINE Integration

Two distinct LINE integrations:
- **LIFF** (`NEXT_PUBLIC_LIFF_ID`): wraps the `/order` route via `LiffProvider` — identifies customers by `line_user_id` from the LIFF context, no PIN required. Set `NEXT_PUBLIC_LIFF_MOCK=true` in `.env.local` to bypass real LINE auth during local dev — **must be unset or `false` in production**. After order submission, the order receipt card is rasterised with `html2canvas` (dynamically imported) and offered as a shareable image. The LIFF customer identity is persisted in a separate HMAC-signed cookie (`allnoodles_order_session`, 30-day TTL) via `src/lib/auth/order-session.ts` — distinct from the admin PIN session.
- **Messaging API** (`LINE_CHANNEL_ACCESS_TOKEN` + `LINE_GROUP_ID`): `src/lib/line/notify.ts` pushes a Flex Message to a LINE group when a new order is submitted; both vars are optional — notification silently skips if unset
- `/api/line/webhook` — utility endpoint to capture `LINE_GROUP_ID` when the OA bot joins a group (logs it to console)

### Environment Variables

Required in `.env.local`:
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SUPABASE_PROJECT_ID` + `SUPABASE_ACCESS_TOKEN` — needed for `npm run gen:types`
- `LOGIN_PIN_PEPPER` — **never change after first deploy** (invalidates all PINs)
- `SESSION_SECRET` — **never change after deploy** (invalidates all sessions)
- `NEXT_PUBLIC_LIFF_ID`

Optional:
- `LINE_CHANNEL_ACCESS_TOKEN` + `LINE_GROUP_ID` — LINE group notifications (silently skipped if unset)
- `NEXT_PUBLIC_SITE_URL` — notification deep-link base (defaults to `http://localhost:3000`)
- `NEXT_PUBLIC_LIFF_MOCK=true` — local dev only, bypasses real LINE LIFF auth

See `.env.example` for the full template.

### Thai Date & Formatting Utilities

- **`getTodayInBangkok()`** in `src/lib/orders/date.ts` — returns `YYYY-MM-DD` for the current date in `Asia/Bangkok`. Use this instead of `new Date()` whenever you need "today" for order/report filtering; also exports `normalizeOrderDate()` for query param sanitisation.
- **`fmtDateTH(iso)`** in `src/lib/utils/date.ts` — converts `YYYY-MM-DD` → `DD/MM/YYYY` with Buddhist Era year (+543). Use this everywhere dates are displayed in UI.
- **`fmtDateTimeTH(isoDateTime)`** — same but includes `HH:mm` in Bangkok timezone (`Asia/Bangkok`).
- **`src/lib/format/baht-text.ts`** — converts a numeric amount to Thai words (e.g. for billing printouts). Use the exported function rather than reimplementing.
- **`sortProductsByCategory(products, categories)`** in `src/lib/products/sort-by-category.ts` — single source of truth for product display order (by category `sort_order`, then Thai name). Apply at the data-fetching layer.

### Web Push Notifications

Optional web push support for order alerts:
- `src/lib/push/web-push.ts` — sends push notifications to stored `push_subscriptions` rows via the `web-push` library
- Requires `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` env vars (generate once with `npm run push:vapid`)
- `/api/push` — subscription management endpoint (save/delete browser push subscriptions)
- `src/components/push/` — client-side push permission UI
- Notifications silently skip if VAPID vars are unset

### Order Window

The LIFF `/order` route enforces a configurable daily order window stored in `organizations.metadata.orderSettings`:
- `src/lib/order-window.ts` — pure logic: `parseOrderWindowSettings()`, `isOrderOpenAtMinutes()`, `isCustomerOrderEditableAtTime()`
- `src/lib/order-window-server.ts` — server-side wrapper that reads the org's current window from Supabase
- Settings editable at `/settings` (admin); defaults: open `00:00`, close `17:00`
- `allowOrderAfterCutoff: true` disables the close-time check entirely

### PWA & Offline

`public/sw.js` caches the app shell and falls back to `/offline` on network failure. `src/app/manifest.ts` defines PWA metadata. `PwaProvider` registers the service worker in production only. Do not modify `sw.js` without testing offline behavior.

## Coding Conventions

- **Style**: 2-space indentation, double quotes, semicolons, path alias `@/*` → `./src/*`
- **Naming**: PascalCase for React components (`CustomerForm.tsx`), camelCase/descriptive lowercase for modules (`authorization.ts`)
- **Icons**: `lucide-react` only
- **Security**: Review `src/proxy.ts` and `src/lib/supabase/*` carefully when touching auth, sessions, or middleware
