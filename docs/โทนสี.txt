
จากโค้ดปัจจุบัน สต็อกถูกเก็บรวมไว้ใน products.stock_quantity และ products.reserved_quantity เพียงก้อนเดียว ดังนั้นการเพิ่มคลังที่สองไม่ควรเพิ่ม column เช่น stock_quantity_province เพราะจะขยายยากและทำให้ logic กระจายหลายจุด

แบบที่แนะนำ
สร้างระบบมาตรฐานแบบ คลังสินค้า + ยอดคงเหลือแยกตามคลัง + ประวัติการเคลื่อนไหว

1. ตารางคลังสินค้า
warehouses
- id
- organization_id
- code
- name
- is_default
- is_active
ข้อมูลเริ่มต้น:

code	name
MAIN	คลังหลัก
PROVINCE	คลังต่างจังหวัด
2. ตารางยอดคงเหลือ
ย้ายยอดสต็อกออกจาก products ไปเก็บในตารางใหม่:

warehouse_inventory
- warehouse_id
- product_id
- stock_quantity
- reserved_quantity
- updated_at
หนึ่งสินค้าในหนึ่งคลังมีได้เพียงหนึ่ง row โดยกำหนด unique constraint:

unique (warehouse_id, product_id)
ตัวอย่าง:

สินค้า	คลังหลัก	คลังต่างจังหวัด
เส้นเล็ก	120	45
บะหมี่	80	30
แนวทางนี้ตรงกับหลัก PostgreSQL จาก Context7: ใช้ foreign keys, unique constraint หลาย column และ transaction เพื่อรักษาความถูกต้องของข้อมูล

การใช้งานสำหรับคนมีอายุ
ไม่ควรให้เลือกคลังทุกครั้ง เพราะจะใช้งานยุ่งยากและเสี่ยงเลือกผิด

ด้านบนของหน้าที่เกี่ยวข้องกับสต็อก ให้มีปุ่มขนาดใหญ่และมองเห็นชัด:

กำลังใช้งาน: คลังหลัก ▼
เมื่อกดจะเลือกได้:

✓ คลังหลัก
  คลังต่างจังหวัด
ระบบจำคลังล่าสุดของผู้ใช้ไว้ เมื่อเปิดหน้าใหม่ยังอยู่คลังเดิม

หน้าหลักยังหน้าตาเหมือนเดิม เปลี่ยนเพียงข้อมูลให้เป็นของคลังที่เลือก ไม่ควรแสดงตัวเลขสองคลังปะปนกันในตารางเดียว ยกเว้นหน้าสรุปสำหรับผู้ดูแล

กฎสำคัญในแต่ละงาน
งาน	วิธีแยกคลัง
รับสินค้าเข้า	ต้องระบุว่าเข้าคลังไหน ค่าเริ่มต้นคือคลังที่กำลังเปิดอยู่
ปรับยอดสต็อก	ปรับเฉพาะคลังที่เลือก
ดูประวัติ	กรองตามคลัง และแสดงชื่อคลังในทุกรายการ
ออเดอร์	ต้องผูกกับคลังต้นทางก่อนตัดสต็อก
ออกใบส่งของ	ใช้คลังเดียวกับออเดอร์
คืนสินค้า	คืนเข้าคลังต้นทางของใบส่งของอัตโนมัติ
ของเสียหรือสูญหาย	ลดจากคลังต้นทางของรายการนั้น
แจ้งเตือนของใกล้หมด	แยกแจ้งเตือนรายคลัง
รายงาน	เลือกดูคลังเดียว หรือดูรวมทุกคลังได้
การโอนสินค้าระหว่างคลัง
เพิ่มเมนู โอนสินค้าระหว่างคลัง โดยแยกจากการปรับยอด เพื่อให้ประวัติตรวจสอบได้

หน้าจอควรมีเพียง:

จากคลัง: คลังหลัก
ไปคลัง:   คลังต่างจังหวัด

สินค้า        จำนวน
เส้นเล็ก      20
บะหมี่        10

[ยืนยันการโอน]
เมื่อยืนยัน ระบบทำสองรายการพร้อมกัน:

คลังหลัก            -20  โอนไปคลังต่างจังหวัด
คลังต่างจังหวัด     +20  รับโอนจากคลังหลัก
ต้องทำใน transaction เดียว หากฝั่งใดฝั่งหนึ่งล้มเหลวให้ยกเลิกทั้งหมด

ประวัติการเคลื่อนไหว
ตาราง inventory_movements เดิมควรเพิ่ม:

warehouse_id
transfer_id
ประเภท movement เพิ่ม:

transfer_out
transfer_in
return
lost
ทุกการเพิ่ม ลด คืน รับเข้า และโอน ต้องผ่าน RPC ฝั่งฐานข้อมูล ไม่ควรแก้ยอดตรงจาก frontend เพราะ PostgreSQL ควร lock row ที่กำลังเปลี่ยนด้วย SELECT ... FOR UPDATE ตามแนวทาง Context7 เพื่อป้องกันยอดผิดเมื่อมีหลายคนทำรายการพร้อมกัน
